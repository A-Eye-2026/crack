안정적인 서비스 운영을 위한
Technical Risk Mitigation

단순한 신고 기능을 넘어, 발생 가능한 운영적·법적 리스크를 아키텍처 로직으로 사전 차단합니다.

정보통신망법 (명예훼손)

우려: 민원 대상 타겟팅 및 비방

자유 게시판 형태로 운영될 경우, "담당자가 일을 안 한다" 등의 악의적인 조롱과 비방이 섞여 명예훼손 이슈가 발생할 수 있습니다.

방어 로직: Free-text Zero

사용자 텍스트 입력창을 아예 배제했습니다.
오직 사진(Vision)과 자동 좌표(EXIF)만으로 동작하여 모욕적 텍스트 개입을 원천 차단합니다.

개인정보 보호법

우려: 민원인 신상 및 사생활 노출

공공 피드에 신고자의 신원이 드러나거나, 사진 원본의 EXIF 메타데이터를 통해 개인의 이동 동선이 노출될 위험이 있습니다.

방어 로직: Data Minimization

자체 DB 내 민감정보 보관을 없애기 위해 Kakao OAuth 인증만 활용하며, 사진의 EXIF 데이터는 서버에서 파싱 후 즉각 파기합니다.

공무집행방해 (위계)

우려: 매크로 및 허위 신고 폭탄

무의미한 밈(Meme)이나 관계없는 사진을 대량으로 전송하여 행정망을 마비시키는 디지털 테러 가능성이 있습니다.

방어 로직: AI Filtering

YOLOv8 + ResNet50 Vision AI가 이미지를 1차 판독하여, 도로 파손 특성이 발견되지 않으면 DB 적재 전 메모리 단에서 Drop (거절) 처리합니다.

왜 민간 신고 앱은 실패하는가?
그리고 CRACK의 돌파구

1. 법적 책임

공식적인 영조물 배상책임 근거로 쓰일 수 있도록 지자체의 법적 기록망에 직접 전달되어야 함

B2G 파이프라인

향후 Open API를 통해 지자체망 다이렉트 Payload 연동을 추진하여 공식 행정망 편입

2. 행정망 연동

폐쇄적 인트라넷 접근의 어려움과 복잡한 작업 지시 시스템(워크플로우) 통합 비용 발생

표준 JSON & Admin

시스템 이식성을 최우선한 포맷 공유 및 지자체 관제 맞춤형 대시보드(Admin) 무상 분리 지원

3. 중복 다중신고

번화가 위 동일 포트홀에 수백 건의 개별 신고가 발생할 경우 데이터 대조 행정력 낭비 초래

자동 클러스터링

EXIF 좌표 기반 분석으로 반경 내 데이터를 자동 병합(Merge)하여 단일 작업지시서 발행

4. 비즈니스 모델(BM)

공익 서비스의 특성과 짧은 체류 시간으로 인해 B2C 수익 및 광고 창출이 거의 불가능

Data B2B 비즈니스

유저 리워드 생태계 구성, 향후 카카오/T맵 등에 실시간 위험구역 Data API 판매 (B2B)

5. 실질적 해결 권한

민간 기업은 아스팔트 공사를 할 수 없으므로 문제의 완전한 해결을 유도하기 힘듦

양방향 통신 피드백

지자체의 '보수 완료' 상태를 즉각 앱 유저에게 실시간 Push 통보하여 체감 효능감 극대화

Phase 1: Hyperparameter & Environment Tuning

214,219장 (225GB) 대규모 정밀 학습을 위한 기반 구축 과정 (v1 ~ v4)

GPU Optimization

단일 그래픽카드(RTX 4060 Ti 12GB) 메모리 한계를 극복하기 위해 배치 사이즈(Batch 16)와 메모리 스왑이 일어나지 않는 최적의 워커(Workers=8) 비율 탐색

Hyperparameters

Mosaic Augmentation(1.0) 등의 데이터 증강 기법 적용 및 100 Epoch 완주 전 과적합을 막는 Early Stopping (Patience=10) 튜닝

Training Metrics

Test Set 없이 Val 10% 할당으로 과적합 모니터링망 구축

  • 총 100 Epoch 완주 수행
  • 훈련(Train): 192,798장 / 에포크
  • 검증(Val): 21,421장 / 에포크
  • 에포크당 시간: 평균 약 44분 소요
  • 100 에포크 누적: 약 74시간(3일) 소요

Phase 2: Big Data Preprocessing

대규모 데이터셋 구축을 위한 고효율 파이프라인 (v5)

Hardlink Architecture

물리적 파일 복사 없이 파일 시스템 상의 '이름'만 추가 생성하는 기법.

[기존] 225GB + 구조화 복사 225GB = 디스크 잔여 용량 고갈 및 상당한 시간 소요

[개선] 추가 용량 0GB, 지연 없는 즉각적인 IO 처리로 훈련 즉시 돌입 보장

Multi-core Processing

파이썬 multiprocessing.Pool 활용 병렬 JSON 전처리.

단일 코어(Single-thread)로 21만 개의 JSON을 파싱할 경우 하루 이상의 시간이 소요되지만, 가용 CPU 전역 코어를 동원하여 수 분 내에 YOLO TXT 정규화 포맷 변환 및 Split 매핑 완료.

Phase 3: Sinkhole Integration (v6)

New Dataset

'Sinkhole' Class Add

치명적 위험 요소인 씽크홀 데이터를 기존 도로 파손에 통합하여 재구성

Method

Transfer Learning & Fine-tuning

v5 최적 가중치를 상속하여 초기 수렴 속도 제고 및 18 에포크 심화 강화 학습
