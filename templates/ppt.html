Database Architecture

TiDB 기반의 관계도(ERD)

PBKDF2 및 솔팅 보안 메커니즘

1. 업계 표준 알고리즘

Werkzeug의 PBKDF2 사용. Django, Flask의 기본값으로 채택된 지극히 검증된 보안 표준입니다. 60만 번의 반복 연산으로 무차별 공격을 무력화합니다.

2. 강력한 단방향 해싱

한 번 변환되면 복호화가 절대 불가능합니다. 서버 관리자조차 사용자의 원문 비밀번호를 알 수 없으며, 대조를 통한 인증 방식으로만 사용됩니다.

3. 솔팅(Salting) 기술

동일한 비밀번호라도 사용자마다 고유한 솔트값을 섞어 다른 해시를 생성합니다. 레인보우 테이블 공격을 완벽히 차단합니다.

  • 보안 정책 우회: 파이썬 설치가 제한된 서버에서도 즉시 구동

2. 행정망 연동

표준 JSON & Admin

3. 중복 다중신고

번화가 위 동일 포트홀에 수백 건의 개별 신고가 발생할 경우 데이터 대조 행정력 낭비 초래

자동 클러스터링

EXIF 좌표 기반 분석으로 반경 내 데이터를 자동 병합(Merge)하여 게시물 중복 방지

4. 비즈니스 모델(BM)

공익 서비스의 특성과 짧은 앱 체류 시간으로 인해 단순 B2C 광고 수익만으로는 자생력 확보가 힘듦

Data B2B 비즈니스

크랙 포인트를 통한 리워드 생태계 구성, 향후 카카오/T맵 등에 Data API 판매 서비스 연계 (B2B)

5. 실질적 해결 권한

민간 기업은 아스팔트 공사를 할 수 없으므로 문제의 완전한 해결을 유도하기 힘듦

해외 벤치마크 — 민간이 먼저 DB를 구축하고, 정부 예산 사업으로 연결되는 구조

[세그먼트 개선] 픽셀 단위 분할 학습 결과, Mask Loss가 초기 5 Epoch 내에 급격히 안정화되었습니다. 단순 사각형(Box) 탐지에서 나아가 파손 부위의 실제 단면적을 계산할 수 있는 기반을 마련했으며, 현재 10/40 Epoch 시점에서도 주요 클래스 경계 정밀도가 우수하게 개선되는 추세입니다.

오분류 원인 분석 (Matrix)

분류 성능 검증: FP/FN 분석을 통한 개선 근거 확보
[데이터 편향 대응] Confusion Matrix의 대각선(True Positives)이 짙게 분포되어 분류 정확도가 높음을 증명합니다. 다만 Minor Crack의 일부가 배경으로 오인되는 경향을 식별하였으며, 이를 보완하기 위해 Darkness Augmentation 전략을 수립하는 근거 데이터로 활용하였습니다.
